## 二、專題目標

本專題以**學校實驗室中常見的電子元件**為主題，建立一個能透過攝影機**即時偵測並框出**電子零件位置與種類的物件偵測系統。系統目標為：

1. 自行收集並**標註**一個涵蓋 **13 種電子元件**的物件偵測資料集。
2. 以自製資料集訓練一個輕量級 YOLOv8 物件偵測模型。
3. 透過筆電 / USB 攝影機即時執行，在畫面上用方框標出元件位置，並標示類別與信心程度。

**偵測類別（13 類）：**
`二極體`、`三用電表`、`工具箱`、`白色LED`、`紅色LED`、`測試線`、`電阻`、`電容`、`電線`、`綠色LED`、`撥線鉗`、`麵包版`、`IC`

## 三、系統設計

### 3.1 資料集收集與標註

- **拍攝方式**：使用手機對每一種電子元件，從**多種角度**拍攝照片（原始格式含 Apple HEIC），共 **223 張**。
- **格式轉換**：以 `preprocess_and_rename.py` 將 HEIC 轉為標準 JPG 並統一命名。
- **邊界框標註**：使用 **LabelImg** 對每張圖以滑鼠框出元件位置，輸出 YOLO 格式標註（`類別編號 中心x 中心y 寬 高`，皆正規化為 0~1）。
- **訓練 / 驗證切分**：以 8:2 自動切分為訓練集 **182 張**、驗證集 **41 張**。

### 3.2 物件偵測方法

| 項目 | 內容 |
|------|------|
| 模型架構 | YOLOv8n（Ultralytics 輕量級單階段物件偵測模型） |
| 訓練方式 | 遷移學習（載入 COCO 預訓練權重後微調） |
| 影像尺寸 | 480 × 480 |
| 訓練世代 | 80 epochs |
| 批次大小 | 16 |
| 運算裝置 | CPU（Intel Core i9-13900H） |

選用 YOLOv8n 而非傳統 Haar 級聯分類器的理由：Haar 每個類別需各自訓練一個模型，13 類就要 13 個，且推論時需逐一掃描、誤報率高；YOLOv8 為**單次推論（single-shot）**即可同時偵測 13 類並輸出位置，對光照、角度的泛化能力更佳，CPU 即可即時執行。

### 3.3 程式架構與執行流程

| 程式 / 檔案 | 功能 |
|------------|------|
| `preprocess_and_rename.py` | 將原始照片（含 HEIC）轉成標準 JPG |
| `label_tool.bat` | 啟動 LabelImg 進行邊界框標註 |
| `build_dataset_det.py` | 整理標註、切分 train/val、產生 YOLO 偵測資料集與 `data.yaml` |
| `train.py` | 載入 YOLOv8n 訓練，完成後將最佳權重複製為 `best.pt` |
| `detect.py` | 開啟攝影機即時偵測，畫出方框並以中文顯示類別與信心值 |

**執行流程：**

```text
preprocess_and_rename.py → label_tool.bat → build_dataset_det.py → train.py → detect.py
      （轉檔）              （框選標註）        （建立資料集）       （訓練）     （即時偵測）
```

## 四、測試結果

### 4.1 訓練結果（驗證集，41 張）

本次訓練（YOLOv8n，80 epochs，imgsz 480，CPU）在驗證集（41 張）的最終成績：

| 指標 | 數值 |
|------|------|
| Precision（精確率） | 88.2% |
| Recall（召回率） | 95.8% |
| **mAP@0.5** | **98.2%** |
| mAP@0.5:0.95 | 70.0% |

訓練過程中 box_loss / cls_loss / dfl_loss 三項損失皆穩定下降，模型逐步收斂。mAP@0.5 高達 98.2%，代表在驗證集上幾乎能正確框出所有元件；而 mAP@0.5:0.95（要求框的位置更精準）為 70.0%，顯示框的貼合度仍有提升空間。

### 4.2 實機測試

以筆電攝影機實際執行 `detect.py`，將各類電子元件置於鏡頭前，畫面會即時以**彩色方框**框出元件位置，並在框上方以**中文**顯示類別名稱與信心百分比（信心低於 40% 不顯示，避免誤報）。

> 展示影片詳見第八節連結。

### 4.3 結果討論

本資料集每類平均僅約 17 張、且背景相近，模型對「實驗室桌面」這類熟悉場景偵測效果良好，但面對全新背景或嚴重遮擋時仍可能漏偵或誤判。改善方向為擴充資料量與背景多樣性（詳見下一節）。

## 五、挑戰與解決方法

| # | 遇到的挑戰 | 解決方法 |
|---|-----------|---------|
| 1 | **中文編碼崩潰**：程式在 Windows 中文命令列（cp950）輸出中文 / emoji 時直接 `UnicodeEncodeError`。 | 程式開頭加 `sys.stdout.reconfigure(encoding='utf-8')`；`.bat` 加 `chcp 65001`，並將批次檔訊息改為純 ASCII 避免 cmd 以 Big5 誤讀。 |
| 2 | **PyTorch 無法載入**：最新版 torch 2.12.1 出現 `WinError 1114（c10.dll 初始化失敗）`。 | 逐步排除（套件 / VC++ / CPU 指令集 / 路徑 / PATH），確認為版本不相容，改鎖定 `torch 2.5.1+cpu` 並寫入 `requirements.txt`。 |
| 3 | **標註工具崩潰**：LabelImg 在 Python 3.11 / PyQt5 下，畫框、滑鼠滾輪、縮放時因座標傳入浮點數而 `TypeError` 閃退。 | 修補 LabelImg 原始碼（`canvas.py` 的 `drawRect/drawLine`、`labelImg.py` 的 `scroll_request / set_zoom / 滾輪縮放`），將座標以 `int()` 轉型。 |
| 4 | **標註檔誤存 + 類別誤標**：LabelImg 把全部 223 個 `.txt` 誤存到單一資料夾，且有 7 張「工具箱」誤選成「三用電表」。 | 撰寫 `build_dataset_det.py`，以「圖片所在資料夾＝正確類別」為基準，依檔名對應回圖片並**自動修正類別編號**，重建為標準 YOLO 偵測資料集。 |
| 5 | **OpenCV 無法顯示中文**：`cv2.putText` 畫中文是亂碼。 | 改用 PIL + 微軟正黑體繪製方框與中文標籤。 |
| 6 | **資料量偏少、背景相近**：易過擬合、對新場景泛化有限。 | （改善方向）增加每類張數、同角度多拍、擴充不同背景與光線。 |

## 九、心得與反思

這次期末專題我選擇的是學校實驗室中常見的電子元件來做主題，目的是達到成功偵測並分類元件。在採樣的時候我盡量用各種角度去拍，讓訓練出來的模型可以認出不同角度的目標；但有一個小問題是，我採集的時候除了多角度之外，應該在同一個角度也多拍一些張數，讓偵測時的準確度更高、更穩定。後面實作程式碼時並沒有遇到太多困難，主要的卡關是中文在編碼上出現的整合問題（命令列輸出與攝影機畫面顯示中文都需要額外處理），其餘大多是後續準確度與穩定度的調整。

這次專題，我體會到「資料品質與數量」對深度學習模型的重要性——即使驗證數字看起來漂亮，實機面對新環境時仍可能因為樣本不足而不穩定，這讓我對「過擬合」有了更具體的理解。另外，這次在標註資料時也遇到工具當機與標註檔放錯位置的狀況，讓我學到資料前處理與驗證的重要性，原來「把資料整理對」本身就是一門功夫。同時我也熟悉了在 Windows 中文環境下開發的眉角，例如 UTF-8 編碼切換與字型繪製。最後希望能在學期初就先公布期末專題內容，讓我們有更充裕的時間規劃與收集資料，因為期末考結束到繳交的時間稍微緊湊了一些。

## 十、結論

本專題成功建立了一套以 YOLOv8n 為核心的電子元件即時物件偵測系統，涵蓋資料收集、邊界框標註、資料集建立、模型訓練到攝影機即時偵測的完整工作流。系統能在 CPU 上即時運作，並以中文方框標示元件位置與種類。過程中實際解決了中文編碼、PyTorch 相容性、標註工具崩潰與標註救援等工程問題。未來可透過擴充資料量與背景多樣性，進一步提升偵測準確度與實用性。
